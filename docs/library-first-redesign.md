# ライブラリ中心構造への再設計

旧プロジェクト（Image Assistant）の派生として、「ライブラリを正とする」構造でクリーンに作り直す。
旧コードは部品（クライアント・プロセス管理など）の移植元として利用する。

## コンセプト

- 旧: チャット＋生成が主役、ライブラリはプロンプト参照用の補助
- 新: **ライブラリがマスターデータ**。生成した画像はすべてライブラリに保存され、
  各画像に動画を紐づけて生成・保存する。画像・動画とも削除可能で、
  気に入ったものを残して育てていく使い方を想定。

## データ構造（フォルダ＝正、SQLite＝キャッシュ）

```
data/library/
  .studio/                      # アプリ用メタデータ（ユーザーフォルダと衝突しない予約領域）
    index.sqlite3               # 検索用インデックス（FTS5 / embedding）
    sequences/
      <seq-id>.json             # シーケンス定義（クリップ参照の並び）
  <ユーザーフォルダ>/            # 自由な入れ子（物理フォルダをそのままミラー）
    <入れ子フォルダ>/
      20260720-143012-a1b2c3/   # 画像アイテムフォルダ（ID = タイムスタンプ+乱数）
        image.png               # 元画像（生成メタデータ付きPNG）
        thumb.jpg               # サムネイル
        meta.json               # プロンプト・seed・パラメータ・タグ・動画一覧
        videos/
          v001.mp4
          v001.thumb.jpg
```

### ルール

- 「アイテムフォルダかどうか」は `meta.json` の有無で判定する
- 画像の削除 = アイテムフォルダごと削除
- 動画の削除 = ファイル削除 + `meta.json` の該当エントリ削除
- フォルダ間の移動 = 物理移動 + インデックス更新
- 書き込み順序は「ファイル → meta.json → インデックス」で統一
  （途中クラッシュしてもフォルダスキャンで復旧できる）
- SQLite は常に再構築可能。「インデックス再構築」機能を用意する
- 一覧・検索は必ず SQLite 経由（数百〜数千件でも快適に）

### meta.json の例

```json
{
  "id": "20260720-143012-a1b2c3",
  "created_at": "2026-07-20T14:30:12+09:00",
  "prompt": "...",

  "seed": 12345,
  "params": { "steps": 28, "cfg": 5.0, "size": "1024x1024" },
  "tags": ["..."],
  "caption": "...",
  "videos": [

    }
  ]
}
```

## UI（3ペイン）

```
┌──────┬────────────┬──────────┐
│ツリー │ サムネイル   │ 生成パネル │
│      │ グリッド     │          │
└──────┴────────────┴──────────┘
```

- **左ペイン**: フォルダツリー（自由な入れ子）。フォルダの作成・リネーム・移動・削除
- **中央ペイン**: 選択フォルダ内の画像グリッド。画像選択でその画像の動画一覧も表示
- **右ペイン**: 選択に応じて切り替わるコンテキストパネル
  - フォルダ選択時 → 画像生成 UI（生成するとそのフォルダに追加）
  - 画像選択 + 「動画を生成」 → 動画生成 UI（生成すると画像の子として追加）

## シーケンスモード（drone-clip-manager 参考）

ライブラリ内の生成動画（クリップ）をつないで 1 本の動画に編集するモード。
[drone-clip-manager](../../drone-clip-manager/) の Phase 2.6（シーケンス）を参考にする。

- **モード切替**: 上部タブで「ライブラリ / シーケンス」を切り替える
- **UI**: シーケンス一覧 + 配置エリア + クリップパレット（ライブラリの動画を検索・絞り込みして配置）
  - 並べ方は線形（ドラッグ並べ替え）を第一段階とする。生成動画は数秒の完結したクリップで
    in/out 指定が不要なため、まず線形で成立する。drone-clip-manager 同様のノードグラフ編集は
    データモデルを流用できる発展課題として残す（着手時に再判断）
- **連続再生**: 並び順にクリップをプレビュー再生（`<video>` 要素の順次切替。生成動画は
  H.264/VP9 想定なので mpv 埋め込みは不要の見込み。必要になれば検討）
- **書き出し**: ffmpeg で 1 本の動画に連結
  - 同一ワークフロー由来でコーデック・解像度・fps が一致する場合は
    **concat demuxer + stream copy（無劣化）**
  - 混在する場合は再エンコードにフォールバック（実行前に ffprobe で検証して UI に明示）
  - BGM 合成（音声上書き + フェードイン/アウト）は発展課題（drone-clip-manager Phase 2.6b と同方針）
- **データ**: `data/library/.studio/sequences/<seq-id>.json` に保存。
  クリップは `{ item_id, video_file }` の参照として持つ
  - 動画・画像が削除されて参照切れになったクリップは UI で「欠落」表示し、個別に取り除ける
  - シーケンス自体も削除可能
- **前提ツール**: ffmpeg / ffprobe（連結書き出しに必要。PATH または設定でパス指定）

## 技術スタック（続用）

- Electron デスクトップ UI + Python サーバー
- 画像生成: SD WebUI Forge / ComfyUI
- 動画生成: ComfyUI image-to-video ワークフロー
- LLM: llama-server（OpenAI 互換 API）

## 旧コードからの移植候補

| モジュール | 用途 |
|---|---|
| `a1111_client.py` / `comfyui_client.py` | Forge / ComfyUI との通信 |
| `sd_process.py` / `comfy_process.py` / `managed_process.py` | 外部プロセス管理 |
| `llm_client.py` / `embedding_client.py` | LLM・embedding 連携 |
| `prompt_parser.py` | PNG メタデータ読み取り |
| `settings_manager.py` | 設定の保存・読み込み |
| `image_library.py` | FTS5 / ベクトル検索まわりの実装参考（構造は新設計に変更） |

## マイルストーン

1. **ライブラリコア**: フォルダスキャン・meta.json 読み書き・SQLite インデックス・再構築。
   CRUD（アイテム作成/削除/移動、動画追加/削除）をサーバー API として実装
2. **3ペイン UI の骨格**: ツリー + グリッド + 右パネル。既存画像の閲覧・整理ができる状態
3. **画像生成**: 右パネルの画像生成 UI → 生成結果が選択フォルダに保存される
4. **動画生成**: 画像選択 → 動画生成パネル → 生成結果が画像に紐づく
5. **検索・LLM 連携**: FTS5 / ベクトル検索、タグ、`{library_context}` 相当の機能移植
6. **シーケンスモード**: クリップ並べ替え・連続再生・ffmpeg 連結書き出し（無劣化 concat →
   混在時は再エンコード）。ノードグラフ化・BGM 合成は発展
